# Rust 代码审核检查清单 (Code Review Checklist)

## 使用说明
在代码审核时，请逐项检查以下内容。每个检查项都应该标记为 ✅ (通过) 或 ❌ (需要修改)。

---

## 1. 安全性检查 (Safety Checks)

### 1.1 内存安全
- [ ] **无不必要的 `unsafe` 代码**
  - 检查是否有可以用安全代码替代的 `unsafe` 块
  - 每个 `unsafe` 块都有详细的安全性注释

- [ ] **正确的所有权管理**
  - 所有权转移明确且合理
  - 没有悬垂指针或无效引用
  - 生命周期参数正确使用

### 1.2 并发安全
- [ ] **线程安全的共享状态**
  - 使用 `Arc<Mutex<T>>` 或 `Arc<RwLock<T>>` 保护共享数据
  - 避免数据竞争和死锁
  - 正确处理异步代码的取消和超时

---

## 2. 错误处理检查 (Error Handling Checks)

### 2.1 错误处理模式
- [ ] **避免 `unwrap()` 和 `expect()` 滥用**
  - 生产代码中使用 `?` 操作符或 `match` 处理错误
  - `expect()` 仅用于确实不可能失败的情况，并有有意义的错误信息
  - `unwrap()` 仅用于测试代码或原型

- [ ] **适当的错误传播**
  - 使用自定义错误类型或 `anyhow`/`thiserror`
  - 错误信息有意义且可操作
  - 错误类型实现了 `std::error::Error` trait

### 2.2 恐慌处理
- [ ] **避免不必要的恐慌**
  - 使用 `Result` 类型而不是 `panic!`
  - 恐慌钩子已正确设置（如果需要）

---

## 3. 代码质量检查 (Code Quality Checks)

### 3.1 函数设计
- [ ] **函数长度合理**
  - 单个函数不超过 100 行
  - 复杂函数已拆分为多个小函数
  - 每个函数职责单一

- [ ] **函数签名清晰**
  - 参数数量合理（建议不超过 5 个）
  - 返回类型明确
  - 泛型参数有意义的约束

### 3.2 代码复杂度
- [ ] **圈复杂度合理**
  - 避免过深的嵌套（建议不超过 4 层）
  - 复杂的条件逻辑已提取为函数
  - 使用 `match` 而不是复杂的 `if-else` 链

### 3.3 命名规范
- [ ] **遵循 Rust 命名约定**
  - 函数和变量使用 `snake_case`
  - 类型和 trait 使用 `PascalCase`
  - 常量使用 `SCREAMING_SNAKE_CASE`
  - 名称具有描述性且不会误导

---

## 4. 性能检查 (Performance Checks)

### 4.1 内存使用
- [ ] **避免不必要的分配**
  - 使用借用而不是克隆（除非必要）
  - 合理使用 `Cow<T>` 处理可能的克隆
  - 避免在循环中进行不必要的分配

- [ ] **合适的数据结构**
  - 根据使用模式选择合适的集合类型
  - 预分配已知大小的容器
  - 使用迭代器而不是索引访问

### 4.2 算法效率
- [ ] **时间复杂度合理**
  - 避免不必要的 O(n²) 算法
  - 缓存重复计算的结果
  - 使用适当的算法和数据结构

---

## 5. 文档检查 (Documentation Checks)

### 5.1 API 文档
- [ ] **公共 API 有文档**
  - 所有公共函数、结构体、枚举有 `///` 文档注释
  - 文档包含使用示例
  - 参数和返回值有说明

### 5.2 代码注释
- [ ] **复杂逻辑有注释**
  - 解释为什么这样做，而不是做什么
  - 标记了 TODO、FIXME、HACK 等
  - 注释与代码保持同步

---

## 6. 测试检查 (Testing Checks)

### 6.1 测试覆盖
- [ ] **核心逻辑有测试**
  - 主要功能有单元测试
  - 边界条件和错误情况有测试
  - 测试名称清晰描述测试内容

### 6.2 测试质量
- [ ] **测试独立且可重复**
  - 测试之间没有依赖关系
  - 测试数据不依赖外部状态
  - 使用合适的断言和错误信息

---

## 7. 依赖和配置检查 (Dependencies & Configuration Checks)

### 7.1 依赖管理
- [ ] **依赖版本合理**
  - 使用语义化版本约束
  - 避免使用 `*` 版本
  - 新增依赖有充分理由

### 7.2 特性配置
- [ ] **特性标志使用合理**
  - 可选功能使用 feature flags
  - 默认特性集最小化
  - 特性组合经过测试

---

## 8. 平台兼容性检查 (Platform Compatibility Checks)

### 8.1 跨平台代码
- [ ] **平台差异处理正确**
  - 使用 `#[cfg()]` 处理平台特定代码
  - 避免硬编码路径和系统调用
  - 提供合理的回退机制

---

## 9. 安全编程检查 (Secure Programming Checks)

### 9.1 输入验证
- [ ] **外部输入已验证**
  - 用户输入经过验证和清理
  - 使用类型系统防止无效状态
  - 避免注入攻击和路径遍历

### 9.2 敏感数据处理
- [ ] **敏感数据保护**
  - 不在日志中记录敏感信息
  - 使用安全的随机数生成
  - 正确处理密码和密钥

---

## 10. 代码风格检查 (Code Style Checks)

### 10.1 格式和风格
- [ ] **代码格式一致**
  - 通过 `rustfmt` 格式检查
  - 通过 `clippy` lint 检查
  - 遵循项目代码风格指南

---
