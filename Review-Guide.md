# Rust 代码审核指南 (Code Review Guide)

## 概述

本指南基于 Rust 社区最佳实践和安全编程原则，为 SeeU-Desktop 项目提供全面的代码审核标准。

## 1. 安全性 (Safety)

### 1.1 内存安全
- **避免不必要的 `unsafe` 代码**
  - 仅在绝对必要时使用 `unsafe`
  - 每个 `unsafe` 块必须有详细的安全性注释
  - 优先使用安全的替代方案

- **所有权和借用检查**
  - 确保所有权转移明确且合理
  - 避免悬垂指针和数据竞争
  - 正确使用生命周期参数

### 1.2 并发安全
- **线程安全**
  - 使用 `Arc<Mutex<T>>` 或 `Arc<RwLock<T>>` 进行共享状态
  - 避免死锁和竞态条件
  - 正确处理异步代码中的取消和超时

## 2. 错误处理 (Error Handling)

### 2.1 Result 类型使用
- **避免 `unwrap()` 和 `expect()`**
  - 在生产代码中使用 `?` 操作符或 `match`
  - `expect()` 仅用于确实不可能失败的情况，并提供有意义的错误信息
  - `unwrap()` 仅用于测试代码或原型

### 2.2 错误传播
- **使用自定义错误类型**
  - 实现 `std::error::Error` trait
  - 提供有意义的错误信息
  - 使用 `anyhow` 或 `thiserror` 简化错误处理

### 2.3 恐慌处理
- **避免恐慌**
  - 使用 `Result` 类型而不是 `panic!`
  - 在 `main` 函数中设置恐慌钩子
  - 记录恐慌信息用于调试

## 3. 代码质量 (Code Quality)

### 3.1 函数设计
- **函数长度**
  - 单个函数不超过 100 行
  - 复杂函数应拆分为多个小函数
  - 每个函数只做一件事

### 3.2 模块化
- **模块组织**
  - 使用 `mod.rs` 组织模块结构
  - 公共 API 应该简洁明了
  - 内部实现细节应该私有

### 3.3 命名规范
- **遵循 Rust 命名约定**
  - 函数和变量使用 `snake_case`
  - 类型和 trait 使用 `PascalCase`
  - 常量使用 `SCREAMING_SNAKE_CASE`
  - 模块使用 `snake_case`

## 4. 性能 (Performance)

### 4.1 内存使用
- **避免不必要的克隆**
  - 使用借用而不是所有权转移
  - 使用 `Cow<T>` 处理可能需要克隆的情况
  - 合理使用 `Box<T>` 和 `Rc<T>`

### 4.2 算法效率
- **选择合适的数据结构**
  - 根据使用模式选择 `Vec`、`HashMap`、`BTreeMap` 等
  - 避免不必要的重复计算
  - 使用迭代器而不是索引访问

## 5. 文档 (Documentation)

### 5.1 代码文档
- **公共 API 文档**
  - 所有公共函数、结构体、枚举必须有文档注释
  - 使用 `///` 进行文档注释
  - 包含使用示例

### 5.2 内联注释
- **复杂逻辑注释**
  - 解释为什么这样做，而不是做什么
  - 标记临时解决方案和技术债务
  - 使用 TODO、FIXME、HACK 等标记

## 6. 测试 (Testing)

### 6.1 单元测试
- **测试覆盖率**
  - 核心业务逻辑必须有单元测试
  - 边界条件和错误情况必须测试
  - 使用 `#[cfg(test)]` 组织测试代码

### 6.2 集成测试
- **端到端测试**
  - 关键功能流程必须有集成测试
  - 使用 `tests/` 目录组织集成测试
  - 模拟外部依赖

## 7. 依赖管理 (Dependencies)

### 7.1 Cargo.toml
- **版本管理**
  - 使用语义化版本
  - 避免使用 `*` 版本
  - 定期更新依赖

### 7.2 特性标志
- **可选功能**
  - 使用 feature flags 控制可选功能
  - 默认启用最小功能集
  - 文档化所有特性

## 8. 平台兼容性 (Platform Compatibility)

### 8.1 跨平台代码
- **条件编译**
  - 使用 `#[cfg()]` 处理平台差异
  - 避免硬编码路径分隔符
  - 使用标准库的跨平台 API

### 8.2 外部系统调用
- **系统依赖**
  - 最小化系统特定代码
  - 提供回退机制
  - 文档化系统要求

## 9. 安全编程 (Secure Programming)

### 9.1 输入验证
- **用户输入**
  - 验证所有外部输入
  - 使用类型系统防止无效状态
  - 避免 SQL 注入和路径遍历

### 9.2 敏感数据
- **数据保护**
  - 避免在日志中记录敏感信息
  - 使用安全的随机数生成器
  - 正确处理密码和密钥

## 10. 代码风格 (Code Style)

### 10.1 格式化
- **使用 rustfmt**
  - 配置 `rustfmt.toml`
  - 在 CI 中检查格式
  - 统一代码风格

### 10.2 Linting
- **使用 clippy**
  - 修复所有 clippy 警告
  - 配置项目特定的 lint 规则
  - 在 CI 中运行 clippy

## 审核流程

1. **自动化检查**
   - 编译通过
   - 所有测试通过
   - rustfmt 格式检查
   - clippy lint 检查

2. **人工审核**
   - 代码逻辑正确性
   - 架构设计合理性
   - 安全性考虑
   - 性能影响评估

3. **文档审核**
   - API 文档完整性
   - 代码注释质量
   - 变更日志更新

4. **测试审核**
   - 测试覆盖率
   - 测试质量
   - 边界条件测试
